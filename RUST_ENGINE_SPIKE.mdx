# Rust Engine Spike and Porting Guide

This document combines the `OSS-2400` spike specification with the Panda-specific Rust porting rules. It is the source of truth for measurement, audit, and porting discipline until the spike concludes.

## Goal

Validate whether the Rust/Oxc engine should proceed beyond setup. This spike must produce hard numbers and an output-equivalence report before production Rust compiler logic is added.

Gate criteria from Linear:

- End-to-end build is at least 10x faster than the current `ts-morph` and PostCSS pipeline on a representative medium app.
- CSS output has zero unexplained diff against the fixture corpus.
- At least one committed Rust contributor is identified.
- No parser/type-checker dependency blocks an Oxc-based extractor.

## Fixture Corpus

The equivalence harness should cover these behavior groups before Rust extraction becomes opt-in:

| Group | Required cases |
| --- | --- |
| Function extraction | `css`, `cva`, `sva`, pattern calls, recipe calls |
| Import matching | named imports, aliases, namespaces, import maps, tsconfig paths |
| JSX extraction | style props, pattern components, recipe components, JSX factory calls |
| Values | strings, numbers, booleans, null, undefined, arrays, nested objects |
| Static resolution | local constants, object spreads, array spreads, property access, element access |
| Conditionals | ternary, logical `&&`, same-value branches, unresolvable branches |
| Tokens | `token()`, `token.var()`, fallback values, aliases |
| Template syntax | tagged template `css`, template literals with static interpolation |
| Framework transforms | Vue, Svelte, Astro-compatible preprocessed content |
| Output | parser JSON, encoder JSON, generated CSS, source ranges |

## Equivalence Rules

- Do not update snapshots silently.
- Classify every diff as:
  - TS engine bug to fix before comparison.
  - Rust bug to fix before proceeding.
  - Intentional improvement that needs documented approval.
  - Blocker.
- Compare normalized JSON for parser/encoder output where order is not semantically required.
- Compare CSS text byte-for-byte unless a documented and approved divergence exists.
- Source ranges must be tested separately because diagnostics and debug output depend on them.

## Type-Checker Dependency Audit

Audit `packages/parser/src` and `packages/extractor/src` before implementing Rust extraction.

Search commands:

```sh
rg "ts-morph|Node\\.|SyntaxKind|getType|getSymbol|getDeclarations|ts-evaluator|resolveModuleName" packages/parser packages/extractor packages/config -g "*.ts"
```

| Area | Current usage | Rust strategy | Fallback allowed? | Notes |
| --- | --- | --- | --- | --- |
| Import declarations | `SourceFile.getImportDeclarations()` | Oxc module AST visitor | No | First Rust slice |
| Syntax type guards | `Node.is*` checks | Oxc AST matches | No | Direct port |
| Same-file constants | Identifier declaration lookup | Oxc visitor plus scope table or `oxc_semantic` | Short term | Needed for common extraction |
| Cross-file constants | Import/source-file traversal | Two-pass export map for included files | Yes | Restrict to safe object/array constants |
| Type literal extraction | TS type nodes and readonly property signatures | Defer or fallback | Yes | Do not implement full checker |
| Arbitrary expression evaluation | `ts-evaluator` | Panda-specific static evaluator | Yes | No general JS execution |
| Config module resolution | `ts.resolveModuleName()` if present | `oxc-resolver` or lightweight resolver | No | Required for parser independence |

## Porting Rule

Preserve Panda's architecture and output first. Make the Rust implementation recognizable as the same compiler pipeline:

```txt
source files -> extract -> encode -> emit -> optimize -> CSS
```

Do not redesign and port in the same PR. If a better architecture is needed, land the behavior-preserving port first, then improve it behind the same equivalence harness.

## Crate Boundaries

| Crate | Responsibility |
| --- | --- |
| `engine` | Orchestrates extract, encode, emit, optimize, and cache |
| `extractor` | Oxc-based source parsing and Panda usage extraction |
| `encoder` | Style usage to deterministic encoded atomic rules |
| `emitter` | Encoded rules to flat CSS strings |
| `optimizer` | `lightningcss` minification and prefixing |
| `cache` | File, config, buildinfo, and rule caches |
| `config` | Serialized config structures for Rust |
| `packages/compiler/crate` | NAPI boundary only |

The NAPI crate should be thin. Compiler logic belongs in `crates/*`.

## Dependency Rules

- Use Oxc crates for JS/TS/JSX/TSX parsing and traversal.
- Use `lightningcss` for CSS optimization.
- Use `serde` for boundary structs.
- Do not add async runtimes for compiler crates.
- Do not expose Oxc AST nodes across the NAPI boundary.
- Do not add broad framework dependencies without a benchmark or design note.

## Comment Markers

Use exact markers so audits can run with `rg`.

```rust
// TODO(port): explain unsupported or uncertain behavior.
// PERF(port): explain a known performance follow-up and benchmark needed.
// SAFETY: explain the invariant that makes an unsafe block valid.
// PORT NOTE: explain intentional reshaping from TypeScript.
```

Rules:

- `TODO(port)` must block default-on Rust unless the case is behind TS fallback.
- `PERF(port)` must have a benchmark or follow-up issue before default flip.
- Every `unsafe` block must have a nearby `SAFETY` comment.
- `PORT NOTE` helps reviewers; it must not hide missing behavior.

## Output Rules

- Parser output must be equivalent before it is faster.
- Class names and hashes are public behavior; do not change them accidentally.
- CSS output is sacred. Every diff must be classified and approved.
- Source ranges are part of diagnostics and debug output; test them explicitly.

## Testing Requirements

Every compiler crate feature needs:

- Rust unit tests for local behavior.
- Node/Vitest boundary tests if data crosses NAPI.
- Fixture equivalence against the current TypeScript engine.
- A benchmark if the feature is on the hot path.

## Unsafe Policy

Avoid `unsafe` unless there is a clear performance or FFI reason. If used:

- Keep the unsafe block small.
- Add a `SAFETY` comment immediately before it.
- Explain ownership, lifetime, and aliasing assumptions.
- Prefer a safe wrapper with tests around the unsafe operation.

## Review Checklist

- Does the PR preserve current output?
- Is every unsupported behavior marked with `TODO(port)` and covered by fallback?
- Are public boundary types JSON-serializable and stable?
- Does Rust stay behind `experimental.engine: "rust"` unless explicitly flipping defaults?
- Are benchmarks updated when the hot path changes?
