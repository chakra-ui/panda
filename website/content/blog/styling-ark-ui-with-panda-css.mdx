[Ark UI](https://ark-ui.com/) is a library of headless React components. It ships the behavior, accessibility, and state
handling for things like checkboxes, dialogs, and menus, but no styles at all. How they look is left entirely to you.

![Styling Ark UI with Panda CSS](/ark-ui-panda.png)

This post is about styling Ark UI with Panda CSS. We'll build a Checkbox and a Dialog, using Panda's tokens for the
look, its conditions for the interactive states, and a slot recipe to keep the Dialog's many parts in one place.

## Why they work well together

Ark UI exposes the state of every component part through `data-*` attributes, mostly under a single `data-state`
attribute that takes values like `checked`, `open`, and `indeterminate`. It also lets every part accept a `className`.

Panda is built to consume exactly that. Its conditions are selectors, and the built-in ones already match Ark UI's
attributes, so `_checked`, `_open`, and `_disabled` work with no extra setup. For components with several parts, slot
recipes keep all of those styles in one definition.

## Setup

This guide assumes you already have Panda set up in a React project. If not, follow the
[Panda installation guide](/docs/installation/cli) first. Then add Ark UI:

```bash
pnpm add @ark-ui/react
```

Ark UI ships for several frameworks; we'll use the React package here, and the `data-state` attributes (along with the
Panda conditions that match them) are framework-agnostic. Imports are scoped per component, so you only pull in what you
use:

```tsx
import { Checkbox } from '@ark-ui/react/checkbox'
import { Dialog } from '@ark-ui/react/dialog'
```

The Checkbox example uses an icon for its check mark, so add an icon set too (any will do):

```bash
pnpm add lucide-react
```

All the styling examples below import from your generated `styled-system` directory.

## Styling your first component: the Checkbox

An Ark UI Checkbox comes in a few parts. `Checkbox.Control` is the box, `Checkbox.Indicator` holds the check mark,
`Checkbox.Label` is the text, and `Checkbox.HiddenInput` keeps it accessible and form-friendly.

Here's the whole thing, styled with Panda's `css` function:

```tsx
import { Checkbox } from '@ark-ui/react/checkbox'
import { CheckIcon } from 'lucide-react'
import { css } from 'styled-system/css'

export function TermsCheckbox() {
  return (
    <Checkbox.Root
      className={css({
        display: 'inline-flex',
        alignItems: 'center',
        gap: '2.5',
        cursor: 'pointer',
        _disabled: { opacity: 0.5, cursor: 'not-allowed' }
      })}
    >
      <Checkbox.Control
        className={css({
          display: 'flex',
          alignItems: 'center',
          justifyContent: 'center',
          width: '24px',
          height: '24px',
          rounded: 'md',
          border: '2px solid',
          borderColor: 'gray.300',
          color: 'white',
          transition: 'background-color 150ms, border-color 150ms',
          _checked: { bg: 'blue.500', borderColor: 'blue.500' },
          _focusVisible: { outline: '2px solid', outlineColor: 'blue.500', outlineOffset: '2px' }
        })}
      >
        <Checkbox.Indicator className={css({ display: 'flex', alignItems: 'center', justifyContent: 'center' })}>
          <CheckIcon size={16} />
        </Checkbox.Indicator>
      </Checkbox.Control>
      <Checkbox.Label className={css({ fontSize: 'md', fontWeight: 'bold' })}>I agree to the terms</Checkbox.Label>
      <Checkbox.HiddenInput />
    </Checkbox.Root>
  )
}
```

[Check the box on StackBlitz](https://stackblitz.com/edit/stackblitz-starters-adefjdhw?file=app%2Ftheme-checkbox.tsx)
and the control fills in and the check mark appears. The interesting part is the `_checked` block on the control. In
plain CSS, `:checked` only applies to real form inputs, like an `<input type="checkbox">`. But `Checkbox.Control` is a
`<div>`, so it can never be `:checked`, however, the checked styles still show up. How?

Panda's `_checked` is broader than the CSS `:checked` pseudo-class. When Panda compiles it, the generated rule also
targets `[data-state="checked"]`, and that attribute is exactly what Ark UI puts on the control when the box is checked.
So, the moment Ark UI sets `data-state="checked"`, your `_checked` styles match and apply.

The conditions you'll reach for most with Ark UI follow the same pattern. Each one already bakes in the `data-*`
selector to match:

- `_checked` matches `[data-state="checked"]`
- `_indeterminate` matches `[data-state="indeterminate"]`
- `_open` matches `[data-state="open"]`
- `_closed` matches `[data-state="closed"]`
- `_disabled` matches `[data-disabled]`
- `_focus` matches `[data-focus]`
- `_hover` matches `[data-hover]`

So most of Ark UI's state model just works, with no config at all.

## Styling a complex component: the Dialog

A Dialog has a lot more moving parts, and it renders them through a portal, so it's a better test of whether a styling
approach holds up:

```tsx
<Dialog.Root>
  <Dialog.Trigger />
  <Portal>
    <Dialog.Backdrop />
    <Dialog.Positioner>
      <Dialog.Content>
        <Dialog.CloseTrigger />
        <Dialog.Title />
        <Dialog.Description />
      </Dialog.Content>
    </Dialog.Positioner>
  </Portal>
</Dialog.Root>
```

You could drop a `css` call on each part, but then the dialog's styles are scattered across the markup, and adding
something like a size variant gets messy fast. Slot recipes are made for this. You define every part's styles in one
place, and the recipe hands you a ready-made class for each one, variants included.

Here's the Panda slot recipe for the dialog, written with `sva` (the slot variant function):

```tsx
// components/ui/dialog-styles.ts
import { sva } from 'styled-system/css'

export const dialog = sva({
  slots: ['trigger', 'backdrop', 'positioner', 'content', 'title', 'description', 'closeTrigger'],
  base: {
    trigger: {
      px: '4',
      py: '2',
      rounded: 'md',
      bg: 'gray.900',
      color: 'white',
      cursor: 'pointer'
    },
    backdrop: {
      position: 'fixed',
      inset: '0',
      bg: 'black/50',
      zIndex: 50,
      _open: { animation: 'fadeIn 200ms ease-out' },
      _closed: { animation: 'fadeOut 200ms ease-in' }
    },
    positioner: {
      position: 'fixed',
      inset: '0',
      zIndex: 50,
      display: 'flex',
      alignItems: 'center',
      justifyContent: 'center',
      p: '4'
    },
    content: {
      position: 'relative',
      bg: 'white',
      rounded: 'lg',
      boxShadow: 'lg',
      width: 'full',
      _open: { animation: 'scaleIn 200ms ease-out' },
      _closed: { animation: 'scaleOut 200ms ease-in' }
    },
    title: {
      fontWeight: 'semibold',
      fontSize: 'lg'
    },
    description: {
      mt: '2',
      fontSize: 'sm',
      color: 'gray.600'
    },
    closeTrigger: {
      position: 'absolute',
      top: '3',
      right: '3',
      cursor: 'pointer',
      color: 'gray.500',
      _hover: { color: 'gray.800' }
    }
  },
  variants: {
    size: {
      sm: { content: { maxWidth: '320px', p: '5' } },
      md: { content: { maxWidth: '400px', p: '6' } }
    }
  },
  defaultVariants: {
    size: 'md'
  }
})
```

A dialog sits in the middle of the screen instead of anchoring to a trigger, so there are no positioning variables to
deal with. The positioner just centers the content, and Ark UI takes care of the focus trap and scroll lock.

The backdrop and content each animate as they open and close, running on the same idea as everything else here. Ark UI
marks the dialog `data-state="open"` while it's showing and `data-state="closed"` on the way out, and it keeps the
elements around long enough for the exit to finish.

Those are the values `_open` and `_closed` already match, so the only new piece is a handful of keyframes:

```ts
// panda.config.ts
import { defineConfig } from '@pandacss/dev'

export default defineConfig({
  theme: {
    extend: {
      keyframes: {
        fadeIn: {
          from: { opacity: 0 },
          to: { opacity: 1 }
        },
        fadeOut: {
          from: { opacity: 1 },
          to: { opacity: 0 }
        },
        scaleIn: {
          from: { opacity: 0, transform: 'scale(0.95)' },
          to: { opacity: 1, transform: 'scale(1)' }
        },
        scaleOut: {
          from: { opacity: 1, transform: 'scale(1)' },
          to: { opacity: 0, transform: 'scale(0.95)' }
        }
      }
    }
  }
})
```

Now wire each slot into the markup. Calling `dialog()` gives you an object of class names keyed by slot, and the
`Portal` comes from Ark UI:

```tsx
import { Dialog } from '@ark-ui/react/dialog'
import { Portal } from '@ark-ui/react/portal'
import { XIcon } from 'lucide-react'
import { dialog } from './dialog-styles'

export function SignInDialog() {
  const styles = dialog({ size: 'md' })

  return (
    <Dialog.Root>
      <Dialog.Trigger className={styles.trigger}>Open Dialog</Dialog.Trigger>
      <Portal>
        <Dialog.Backdrop className={styles.backdrop} />
        <Dialog.Positioner className={styles.positioner}>
          <Dialog.Content className={styles.content}>
            <Dialog.CloseTrigger className={styles.closeTrigger}>
              <XIcon />
            </Dialog.CloseTrigger>
            <Dialog.Title className={styles.title}>Welcome back</Dialog.Title>
            <Dialog.Description className={styles.description}>Sign in to your account to continue.</Dialog.Description>
          </Dialog.Content>
        </Dialog.Positioner>
      </Portal>
    </Dialog.Root>
  )
}
```

## A cleaner API with createStyleContext

Threading `styles.trigger`, `styles.content`, and the rest through the markup works, but repeating it every time you
reach for the dialog gets old. Panda has a nicer option: a helper called `createStyleContext`. Give it your slot recipe
once and it hands back components with the styles already attached, so you stop wiring up class names by hand.

`createStyleContext` comes from `styled-system/jsx`, which Panda only generates once `jsxFramework` is set. If you
haven't set it yet, add it to your `panda.config.ts`:

```ts
// panda.config.ts
import { defineConfig } from '@pandacss/dev'

export default defineConfig({
  jsxFramework: 'react'
})
```

`createStyleContext` gives you a few wrappers. The two you need here are `withRootProvider`, for the part that owns the
context (the root), and `withContext` for every other part. You wrap each Ark UI part once, tell it which slot it maps
to, and re-export the whole set as a single `Dialog` object. `Portal` carries no styles, so it passes straight through:

```tsx
// components/ui/dialog.tsx
import { Dialog as BaseDialog } from '@ark-ui/react/dialog'
import { Portal } from '@ark-ui/react/portal'
import { createStyleContext } from 'styled-system/jsx'
import { dialog } from './dialog-styles'

const { withRootProvider, withContext } = createStyleContext(dialog)

export const Dialog = {
  // Root renders no DOM element, so it only provides context
  Root: withRootProvider(BaseDialog.Root),
  Trigger: withContext(BaseDialog.Trigger, 'trigger'),
  Backdrop: withContext(BaseDialog.Backdrop, 'backdrop'),
  Positioner: withContext(BaseDialog.Positioner, 'positioner'),
  Content: withContext(BaseDialog.Content, 'content'),
  Title: withContext(BaseDialog.Title, 'title'),
  Description: withContext(BaseDialog.Description, 'description'),
  CloseTrigger: withContext(BaseDialog.CloseTrigger, 'closeTrigger'),
  Portal
}
```

Now the markup is clean, with no class names in sight, and the `size` variant flows down through context:

```tsx
import { Dialog } from './components/ui/dialog'
import { XIcon } from 'lucide-react'

export function SignInDialog() {
  return (
    <Dialog.Root>
      <Dialog.Trigger>Open Dialog</Dialog.Trigger>
      <Dialog.Portal>
        <Dialog.Backdrop />
        <Dialog.Positioner>
          <Dialog.Content>
            <Dialog.CloseTrigger>
              <XIcon />
            </Dialog.CloseTrigger>
            <Dialog.Title>Welcome back</Dialog.Title>
            <Dialog.Description>Sign in to your account to continue.</Dialog.Description>
          </Dialog.Content>
        </Dialog.Positioner>
      </Dialog.Portal>
    </Dialog.Root>
  )
}
```

[See the createStyleContext version on StackBlitz](https://stackblitz.com/edit/stackblitz-starters-adefjdhw?file=app%2Fdialog-with-context.tsx).

A couple of things to keep in mind. If your root does render an element you want to style, reach for `withProvider`
instead of `withRootProvider` and give it the slot name. And every wrapped part takes an `unstyled` prop, handy when you
want a single part to skip the recipe styles.

## Get started

Ark UI gives you accessible, unstyled components, and Panda gives you a tidy way to style them. You style each part
through its `className`, and the states you care about (checked, open, disabled, and the rest) work out of the box,
because Panda's conditions already match the `data-state` attributes Ark UI sets. For the multi-part ones, a slot recipe
keeps every part's styles together, and `createStyleContext` turns that recipe into a set of components you can reuse
anywhere without touching class names.

To go deeper, see the Panda docs on [recipes](/docs/concepts/recipes), [slot recipes](/docs/concepts/slot-recipes),
[conditional styles](/docs/concepts/conditional-styles), and [JSX style context](/docs/concepts/jsx-style-context). On
the Ark UI side, the [Checkbox](https://ark-ui.com/react/docs/components/checkbox) and
[Dialog](https://ark-ui.com/react/docs/components/dialog) docs are the references to keep open while you build.
